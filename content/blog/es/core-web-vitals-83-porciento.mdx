## Punto de partida

Cuando ejecuté Lighthouse por primera vez sobre las páginas clave de Torre.ai, los números eran preocupantes. El LCP rondaba los 4.2 segundos, el CLS era errático y el FID apenas pasaba el umbral. Necesitábamos un enfoque sistemático.

## La auditoría

Antes de optimizar cualquier cosa, instrumentamos todo. El Real User Monitoring (RUM) a través de CloudWatch nos dio datos de campo que las pruebas sintéticas no capturan:

```typescript
// Reportador custom de Web Vitals
function reportVital(metric: WebVitalMetric) {
  const body = {
    name: metric.name,
    value: metric.value,
    rating: metric.rating,
    navigationType: getNavigationType(),
    pathname: window.location.pathname,
  };
  navigator.sendBeacon('/api/vitals', JSON.stringify(body));
}
```

## Largest Contentful Paint (LCP)

La mayor ganancia vino de tres cambios:

1. **CSS crítico inline** — extrajimos los estilos above-the-fold y los pusimos inline en el head
2. **Optimización de imágenes** — cambiamos a WebP/AVIF con atributos `srcset` y `sizes` apropiados
3. **Estrategia de carga de fuentes** — `font-display: swap` con archivos WOFF2 precargados

## Cumulative Layout Shift (CLS)

El CLS fue el problema más escurridizo. El contenido dinámico causaba saltos de layout que frustraban a los usuarios:

- Dimensiones explícitas en todas las imágenes y embeds
- Espacio reservado para componentes async con skeletons
- Carga de fuentes web más temprana para evitar el reflow del texto

## Los resultados

Después de tres semanas de optimización iterativa:

- **LCP**: 4.2 s → 1.1 s (74% de mejora)
- **CLS**: 0.28 → 0.02 (93% de mejora)
- **FID**: 180 ms → 45 ms (75% de mejora)
- **Puntuación general**: 83% de mejora en todos los Core Web Vitals

¿La lección clave? Medir primero, optimizar después, y validar siempre con datos de usuarios reales.
