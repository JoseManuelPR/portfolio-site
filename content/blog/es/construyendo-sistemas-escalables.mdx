## El desafío

Cuando me uní a Torre.ai, la plataforma crecía rápido. Necesitábamos sistemas que pudieran manejar miles de usuarios concurrentes manteniendo tiempos de respuesta por debajo del segundo. Este post comparte los patrones que funcionaron.

## Patrón 1: arquitectura orientada a eventos

En vez de request-response síncrono para todo, movimos los caminos críticos a patrones event-driven. Esto desacopló los servicios y permitió escalarlos de forma independiente.

```typescript
// Antes: acoplamiento fuerte
await notificationService.send(userId, message);
await analyticsService.track(event);

// Después: event-driven
eventBus.emit('user.action', { userId, event, message });
```

## Patrón 2: caching estratégico con ISR

Para contenido que cambia con poca frecuencia, Incremental Static Regeneration (ISR) en Next.js fue transformador. Redujimos la carga del servidor en un 70% manteniendo el contenido fresco.

```typescript
export const revalidate = 3600; // Revalida cada hora

export default async function Page() {
  const data = await fetchData();
  return <Component data={data} />;
}
```

## Patrón 3: observabilidad primero

No puedes optimizar lo que no puedes medir. AWS CloudWatch se convirtió en nuestros ojos dentro del sistema. Cada camino crítico fue instrumentado desde el día uno.

## Conclusiones

1. **Empieza simple, escala con intención** — la optimización prematura es real
2. **Mide todo** — CloudWatch, métricas custom, KPIs de negocio
3. **Diseña para fallos** — toda llamada externa eventualmente fallará
4. **Cachea agresivamente** — pero con una estrategia clara de invalidación

¿El resultado? 83% de mejora en Core Web Vitals y una plataforma que maneja el tráfico pico sin despeinarse.
