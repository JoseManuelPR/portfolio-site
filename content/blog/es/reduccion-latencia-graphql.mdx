## El problema

La API REST de Kiui había crecido de forma orgánica. Una sola página del dashboard requería 7 llamadas a la API, y cada una retornaba más datos de los que el frontend necesitaba. Payload total: 340 KB. Tiempo hasta interactivo: 3.2 segundos.

## Por qué GraphQL

GraphQL no fue una bala de plata — fue la herramienta correcta para nuestros problemas específicos:

1. **Over-fetching**: los endpoints REST retornaban objetos completos cuando necesitábamos 3 campos
2. **Under-fetching**: el dashboard necesitaba datos de 7 endpoints, causando requests en cascada
3. **Type safety**: los desarrolladores frontend adivinaban la forma de las respuestas

## La estrategia de migración

No reescribimos todo. En su lugar, envolvimos los servicios REST existentes detrás de un gateway GraphQL:

```typescript
const resolvers = {
  Query: {
    dashboard: async (_, { businessId }, context) => {
      // Fetch en paralelo desde los servicios existentes
      const [revenue, orders, inventory] = await Promise.all([
        revenueService.getSummary(businessId),
        orderService.getRecent(businessId, { limit: 10 }),
        inventoryService.getLowStock(businessId),
      ]);

      return { revenue, orders, inventory };
    },
  },
};
```

## Resultados

Después de migrar las 5 páginas con más tráfico:

- **Tamaño de payload**: 340 KB → 45 KB (87% de reducción)
- **Llamadas a la API por página**: 7 → 1
- **Latencia**: 35% de reducción end-to-end
- **Velocidad de desarrollo**: el equipo frontend dejó de esperar cambios en endpoints del backend

## Lecciones aprendidas

- Empieza con las páginas de mayor tráfico, no con toda la API
- El patrón DataLoader es esencial para evitar queries N+1
- El diseño del schema es diseño de API — invierte tiempo en hacerlo bien
- Monitorea el rendimiento de los resolvers desde el día uno
