## El Problema

La API REST de Kiui habia crecido organicamente. Una sola pagina de dashboard requeria 7 llamadas API, cada una retornando mas datos de los que el frontend necesitaba. Payload total: 340KB. Tiempo a interactivo: 3.2 segundos.

## Por Que GraphQL

GraphQL no fue una bala de plata — fue la herramienta correcta para nuestros problemas especificos:

1. **Over-fetching**: los endpoints REST retornaban objetos completos cuando necesitabamos 3 campos
2. **Under-fetching**: el dashboard necesitaba datos de 7 endpoints, causando requests en cascada
3. **Type safety**: los desarrolladores frontend adivinaban las formas de las respuestas

## La Estrategia de Migracion

No reescribimos todo. En su lugar, envolvimos los servicios REST existentes detras de un gateway GraphQL:

```typescript
const resolvers = {
  Query: {
    dashboard: async (_, { businessId }, context) => {
      // Fetch paralelo desde servicios existentes
      const [revenue, orders, inventory] = await Promise.all([
        revenueService.getSummary(businessId),
        orderService.getRecent(businessId, { limit: 10 }),
        inventoryService.getLowStock(businessId),
      ]);

      return { revenue, orders, inventory };
    },
  },
};
```

## Resultados

Despues de migrar las 5 paginas con mas trafico:

- **Tamano de payload**: 340KB → 45KB (87% de reduccion)
- **Llamadas API por pagina**: 7 → 1
- **Latencia**: 35% de reduccion end-to-end
- **Velocidad de desarrollo**: el equipo frontend dejo de esperar cambios en endpoints del backend

## Lecciones Aprendidas

- Empieza con las paginas de mayor trafico, no con toda la API
- El patron DataLoader es esencial para evitar queries N+1
- El diseno del schema es diseno de API — invierte tiempo en hacerlo bien
- Monitorea el rendimiento de los resolvers desde el dia uno
